# دليل إعداد الأتمتة والوسائط — Dilmi TV

## 1. Cloudinary (CDN مجاني للصور) — أولوية عالية، يحل مشكلة موجودة فعلاً

### لماذا هذا يحل مشكلتك بالضبط؟
ذكرت "مشاكل في روابط الصور" — السبب التقني الدقيق: خطة Render المجانية
تستخدم تخزيناً **مؤقتاً وغير دائم** لأي ملف تُرفعه من `/admin/` (شعارات،
صور أخبار...). كل نشر جديد (`deploy`) أو إعادة تشغيل تلقائي **يمحو هذه
الصور بالكامل**، رغم بقاء السجل نفسه في قاعدة البيانات — فتظهر الصورة
"مكسورة" (رابط لملف لم يعد موجوداً).

### خطوات الإعداد (5 دقائق، مجاني بالكامل)
1. أنشئ حساباً على https://cloudinary.com (الخطة المجانية سخية: 25GB تخزين)
2. من لوحة التحكم الرئيسية، انسخ 3 قيم من قسم "Product Environment Credentials":
   - `Cloud Name`
   - `API Key`
   - `API Secret`
3. أضفها كمتغيرات بيئة على Render:
   ```
   CLOUDINARY_CLOUD_NAME=...
   CLOUDINARY_API_KEY=...
   CLOUDINARY_API_SECRET=...
   ```
4. أعد النشر (`Clear build cache & deploy`)

**بعدها**: أي صورة جديدة تُرفعها من `/admin/` تُخزَّن على Cloudinary
تلقائياً وتبقى دائمة إلى الأبد، بغض النظر عن عدد مرات النشر.

⚠️ **الصور القديمة المرفوعة قبل هذا التحديث** (إن وُجدت وما زالت تظهر
حالياً) لن تنتقل تلقائياً — أعد رفعها من `/admin/` مرة واحدة بعد التفعيل.

## 2. API-Football (المزامنة التلقائية للمباريات)

### خطوات الاشتراك
1. أنشئ حساباً على https://rapidapi.com
2. ابحث عن **"API-FOOTBALL"** (وليس أي API آخر بنفس الاسم تقريباً —
   تأكد من الناشر `api-sports`)
3. اشترك بالخطة المجانية للتجربة (100 طلب/يوم تقريباً — كافية لتجربة
   المزامنة اليدوية، غير كافية لأتمتة كاملة لاحقاً)
4. من صفحة الـ API، انسخ **X-RapidAPI-Key**
5. أضفه كمتغير بيئة على Render:
   ```
   API_FOOTBALL_KEY=...
   ```

### كيفية الاستخدام الآن (يدوياً، بدون Celery)
من تبويب **Shell** على Render (إن كانت خطتك تدعمه)، أو محلياً على جهازك
إن كان متصلاً بنفس قاعدة بيانات Neon:
```bash
# يجلب كل مباريات اليوم من كل البطولات
python manage.py sync_matches

# يجلب فقط المباريات المباشرة الآن + أحداثها (أهداف/بطاقات)
python manage.py sync_matches --live-only
```

### بديل مؤقت للتلقائية الكاملة (بدون الحاجة لـ Celery/Redis الآن)
بما أن خطة Render المجانية لا توفر Shell دائماً، الخيار العملي الأسهل
الآن: أضف رابط API خاص محمي (مثلاً `/api/internal/sync-trigger/`) يُشغّل
نفس منطق `sync_matches` عند استدعائه، ثم استخدم خدمة **مجانية** مثل
[cron-job.org](https://cron-job.org) لاستدعاء هذا الرابط كل 5-10 دقائق
تلقائياً — هذا "تشغيل دوري" حقيقي دون أي تكلفة أو حاجة لـ Celery/Redis.
**أخبرني إن أردت هذا الحل تحديداً، يمكن إضافته بسرعة كخطوة تالية.**

## 3. المرحلة القادمة: Celery + Redis + Django Channels

عندما تكون مستعداً للترقية لخطة مدفوعة (على Render أو منصة أخرى)،
الانتقال سيكون **سلساً جداً** بفضل الفصل المعماري الحالي:

- `core/services/match_sync.py` يحتوي **كل منطق المزامنة** بمعزل تام عن
  "من يستدعيه" — تحويله لمهمة Celery يعني فقط:
  ```python
  # core/tasks.py (جديد وقتها)
  from celery import shared_task
  from core.services.match_sync import sync_live_fixtures_with_events

  @shared_task
  def sync_live_matches_task():
      sync_live_fixtures_with_events()
  ```
  ثم جدولتها عبر `celery-beat` كل دقيقة — **دون تعديل حرف واحد** في
  `match_sync.py` نفسه.

- Django Channels للـ WebSockets يحتاج طبقة بنية تحتية منفصلة (ASGI
  server + Redis كـ Channel Layer) — سنبنيها فوق الـ Models الحالية
  دون أي إعادة هيكلة، فقط إضافة `consumers.py` جديد يبث تحديثات `Match`
  فور حفظها (عبر Django Signals).

أخبرني عندما تقرر الترقية، وسأنفّذ هذا الجزء بالكامل حينها.
