# الاعتماد الكامل على Neon — نهاية تضارب SQLite/PostgreSQL

## 1. ماذا تغيّر بالضبط

### أ) `settings.py`
`DATABASE_URL` أصبح **إلزامياً في كل بيئة** (وليس فقط الإنتاج). إذا لم
يكن موجوداً، **يتوقف المشروع فوراً** برسالة خطأ عربية واضحة، بدل
الاستمرار بصمت على `db.sqlite3` محلي منفصل عن بيانات Neon الحقيقية —
وهذا بالضبط كان سبب "التضارب" الذي وصفته.

### ب) `models.py` — تحصين حقيقي على مستوى النموذج
`League.save()` و `Team.save()` الآن **يمنعان حفظ اسم فارغ من الأساس**
(يستبدلانه تلقائياً بـ "بطولة غير معروفة (#ID)" مثلاً)، بغض النظر عن
مصدر البيانات (مزامنة API، إدخال يدوي، استيراد مستقبلي). هذا أقوى بكثير
من التحصين في `match_sync.py` فقط، لأنه يحمي **كل** طريق ممكن للبيانات
للوصول لقاعدة البيانات، وليس المزامنة فقط.

`Match.__str__` و`competition_display` الآن محصَّنان بالكامل ضد أي قيمة
`None` أو تنسيق تاريخ تالف — لن يرميا استثناءً مطلقاً.

### ج) `admin.py`
كل حقول `list_display` التي قد تعتمد على بيانات خارجية الآن تمرّ عبر
دوال آمنة (`safe_name`, `safe_match_title`, `safe_competition`) تلتقط
أي استثناء داخلياً وتعرض نصاً بديلاً بدل إسقاط الصفحة بالكامل.

### د) أمر التنظيف يعمل تلقائياً الآن
`cleanup_sync_data --apply` أصبح جزءاً من `Build Command` نفسه (راجع
القسم 3 أدناه) — يُشغَّل **تلقائياً عند كل نشر جديد**، فينظّف أي سجل
باسم فارغ فوراً دون تدخل يدوي.

## 2. الإعداد المحلي الجديد (إلزامي الآن)

لم يعد بإمكانك تشغيل المشروع محلياً بدون `DATABASE_URL`. من PowerShell:

```powershell
$env:DATABASE_URL="نفس-الرابط-الموجود-في-Render-Environment"
python manage.py runserver
```

هذا يضمن أنك **دائماً** تعمل على نفس بيانات Neon الحقيقية محلياً
وفي الإنتاج — لا مزيد من التضارب.

## 3. Build Command الدقيق على Render (حدّثه الآن)

اذهب إلى: خدمتك ← **Settings** ← **Build Command**، واستبدله بـ:

```
pip install -r requirements.txt && python manage.py collectstatic --noinput && python manage.py migrate && python manage.py cleanup_sync_data --apply && python manage.py create_admin
```

**الترتيب منطقي وإلزامي:**
1. تثبيت المكتبات
2. تجميع الملفات الثابتة
3. **`migrate`** — يُحدّث مخطط Neon تلقائياً في كل نشر (هذا يجيب على
   سؤالك: نعم، Render سينفّذ `migrate` على Neon تلقائياً في كل مرة
   تدفع كوداً جديداً لـ GitHub، دون أي حاجة لـ Shell)
4. **`cleanup_sync_data --apply`** — تنظيف تلقائي للسجلات المعطوبة
5. `create_admin` — تأكيد وجود حساب المدير (آمن، لا يكرر شيئاً)

**Start Command** يبقى دون تغيير:
```
gunicorn dilmi_tv_backend.wsgi:application
```

## 4. تحقّق نهائي بعد النشر

راقب سجل البناء (Build Logs) للتأكد من ظهور هذه الأسطر بالترتيب:
```
Applying core.XXXX... OK
لا توجد سجلات بطولة معطوبة. ✓   (أو: تم حذف X سجل بطولة معطوب)
تم إنشاء حساب المدير "..." بنجاح.  (أو: موجود بالفعل)
```
